# Лабораторна 1: UML Діаграми 📊

## Опис завдання

Лабораторне завдання складається з чотирьох частин, кожна з яких реалізовується як окрема UML діаграма.

## 🎯 Загальні рекомендації

### 1. Use Case діаграма
- Для першого завдання при створенні діаграми використання слід визначити та представити основних акторів

### 2. Діаграма класів
- Слід використовувати можливості ООП:
  - Класи
  - Успадкування
  - Поліморфізм
  - Інкапсуляція
- **Примітка:** Зазвичай діаграма класів служить для представлення статичної структури моделі системи в цілому, однак для цілей цієї лабораторної роботи намалюйте лише ту її частину, яка стосується поставленої задачі

### 3. Діаграма активності
- Використання таймерів на діаграмі активності слід робити лише при потребі

## ✅ Завдання для варіанту Spotify

### 1️⃣ Use Case діаграма
**Намалюйте use-case діаграму для:**
- Реєстрації нового користувача
- Створення плейлиста

### 2️⃣ Діаграма класів
**Намалюйте діаграму класів для представлення множини об'єктів, необхідних для:**
- Створення плейлиста
- Програвання плейлиста
- Збереження статистики прослуховування пісень

### 3️⃣ Діаграма активності
**Намалюйте діаграму активності для зняття оплати:**
- В залежності від типу підписки користувачем
- ✅ Відобразити **всі типи підписки** (FREE, PREMIUM, STUDENT)

### 4️⃣ Sequence діаграма
**Намалюйте sequence діаграму, яка показує:**
- Як зчитується інформація про плейлист користувача
- Як записується статистика про прослуховування кожної пісні

# Лабораторна 2: 3-Рівнева Архітектура 🏗️

## Опис завдання

Реалізуйте серверну частину аплікації, котра складається з **трьох рівнів**:

1. **Рівень доступу до даних (DAL)** - Data Access Layer
2. **Рівень бізнес-логіки (BLL)** - Business Logic Layer  
3. **Презентаційний рівень (PL)** - Presentation Layer

## 🔗 Архітектурні вимоги

### Зв'язок між рівнями
- Реалізуйте із застосуванням **інтерфейсів**
- Класи рівня бізнес-логіки мають використовувати інтерфейси рівня доступу до даних, **а не імплементацію**
- Використовуйте шаблони:
  - **Інверсія управління (IoC)**
  - **Впровадження залежностей (Dependency Injection)**

### Презентаційний рівень
- На даний момент **не виконує жодної логіки**
- Представлений **лише інтерфейсами**

## 📋 Вимоги до кожного рівня

### 1. Рівень доступу до даних (DAL)
- Реалізуйте з використанням **ORM фреймворку**
- Для заповнення бази даних, створеної в лабораторній 1.b
- Реалізуйте **зчитування даних з .csv файлу**
- Обробка дублікатів та валідація даних

### 2. Рівень бізнес-логіки (BLL)
- Здійснює виклик рівня доступу до даних для:
  - Вичитування даних з файлу
  - Створення необхідних моделей
  - Заповнення бази даних
- Викликає рівень доступу до даних для збереження інформації в БД

### 3. Дані та тестування
- ✅ **Всі дані мають міститись в одному файлі**
- При завантаженні даних в БД слід реалізувати необхідну логіку:
  - Коректне збереження даних в таблиці
  - Обробка дублікатів
  - Валідація
- 📊 **Файл має містити мінімально 1000 рядків**
- 🔧 Для створення файлу слід створити окремий **модуль/клас**, який запускається з командного рядка

# Лабораторна 3: MVC Веб-додаток 🌐

## Опис завдання

На основі шаблону **MVC** створіть веб-додаток, який за запитом користувача візуалізуватиме дані з предметної області описаної в **Лабораторній роботі №1** (згідно отриманого варіанту).

## ✅ Умови реалізації

### 1️⃣ Основна сутність
- Визначте, що буде основною сутністю вашої предметної області
- *Приклад:* Для інтернет-магазину - це товар

### 2️⃣ Контролер(и)
- Реалізуйте Контролер(и), з відповідними методами дії
- Методи можуть відображати інформацію про сутність вашої предметної області

### 3️⃣ Модель
- Створіть Модель, що описує логіку даних вашої предметної області
- Модель взаємодіє із базою даних
- База даних наповнюється відповідними тестовими даними

### 4️⃣ CRUD операції
- ✅ Реалізуйте можливість **додавати** дані
- ✅ Реалізуйте можливість **редагувати** дані
- ✅ Реалізуйте можливість **видаляти** дані

### 5️⃣ Представлення (View)
- Вибірка даних із Моделі відображається за допомогою **Представлення**
- Подається у вигляді **HTML-сторінки**

### 6️⃣ Business Logic Layer
- ✅ Вичитка даних відбувається з використанням **класів рівня бізнес-логіки**

## 📖 Корисне посилання

- [Metanit - MVC 5](https://metanit.com/sharp/mvc5/)

# Лабораторна 4: GoF-паттерн Стратегія 🎯

## «Реалізація GoF-паттерну Стратегія»

## Опис завдання

Код, який виводить рядки в консоль, має бути відділений від коду, що робить вичитку даних з файлу.

## ✅ Вимоги

### Організація коду
- Код виводу в консоль має бути організований з використанням **паттерну Стратегія**
- Дозволяти **переключити вивід** з консолі на запис у **Kafka**

### Гнучкість конфігурації
- **Мінімальні зміни в коді**
- Переключення через **зміни в конфігураційних файлах**
- **Без змін в самому коді**

## 📝 Контекст

**P.S.** Контекст завдання пов'язаний із ЛР №7 із БД.
